Checking in an Object

The SM can pass a message which tells the FE to create an image in the workspace of an object which exists in the SM. The message has the form



#1 objectID typeID iconID name dependencies properties viewStructure definition

An objectID is a unique positive integer assigned to the object by the SM. It is the only definitive reference to the object shared by the FE and SM.

A typeID is a positive integer assigned to the object by the SM. It is used to identify what actions may be taken on a set of objects.

An iconID is one of the symbols: group, subgroup, elt, map, enumerator, problem.

A name is the text which is to be displayed beneath the icon in the workspace. It must be delimited by {} if it contains whitespace.

The dependencies is a list of the form {objectID*}. In this case objectID is not a keyword identifier.

The properties is a list of the form {{key datum}+}, where key and datum are strings matching '[a-zA-Z0-9(), ]+'. By convention, all spaces are stripped out by the FE. The key-datum pairs are stored by the FE on behalf of the SM for later use. The purpose is to allow the SM to endow the FE with certain knowledge (about objects, in this case), which the FE can use as directed without knowing anything about it. See §2.3.1.2 for how to use this stored information.

A viewStructure has the form: {viewBuilder viewParameters}. A viewBuilder is the name of a predefined view window builder (§2.3.3). The viewParameters are the parameters expected by the viewBuilder. The viewBuilder may be {} if no view is defined.

The definition is supplied only when the name is the empty string, and the object is not a problem object. This is the case when a new object is constructed by the SM in response to an end user query.
