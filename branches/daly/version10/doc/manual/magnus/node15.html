Testing Procedures

This section describes the testing procedures for MAGNUS. Roughly, one may distinguish two levels of tests.

  • General `external' tests, which test the behaviour of the system or its components with respect to their external calling interfaces, viz. for classes their public methods.

  • Specific low-level tests, which may be more specialised in their way of verifying the internal functioning of (functions of) the system. For instance, if a SetOf class constructs hash tables, the implementer may desire or need to devise specific tests which verify the internal layout of the tables.

The external tests (with as the need arises their associated test data sets) are mandatory, as these are necessary to know whether the system functions at all. Low level tests are a function of their necessity in the course of implementation and not an end in themselves. For these, the guiding principle is to write them only as need dictates, for instance when debugging; but if writing some, then to keep these for possible reuse.

The aim is to

  • build up a set of tests for MAGNUS concurrently to the development of the system;

  • organise this set as a reusable library of standard tests for the system.

The reasons are the obvious:

  • Tests are onerous to devise and implement;

  • Better results are obtained by testing from the beginning onwards;

  • Tests need not be rebuilt from scratch to localise later bugs;
